Overview

A comprehensive ESLint configuration for React and TypeScript projects with integrated Prettier support.

Features

  • Type-safe linting with TypeScript ESLint
  • Prettier integration for consistent code formatting
  • React support with hooks and JSX rules
  • Accessibility checks with jsx-a11y
  • Import sorting and organization
  • Best practices from Airbnb style guide

Installation

# npm
npm install -D @ytvee-dev/eslint-config-react

# yarn
yarn add -D @ytvee-dev/eslint-config-react

# pnpm
pnpm add -D @ytvee-dev/eslint-config-react

Configuration

Create eslint.config.mjs in your project root: 

import reactConfig from "@ytvee-dev/eslint-config-react/configs/react";

export default [...reactConfig];

Profiles

Base Profile

Fundamental ESLint rules, TypeScript type-checking, and Prettier formatting. 

import baseConfig from "@ytvee-dev/eslint-config-react";

export default [...baseConfig];

React Profile

Base profile + React-specific rules, hooks, and accessibility checks. 

import reactConfig from "@ytvee-dev/eslint-config-react/configs/react";

export default [...reactConfig];

Strict Profile

Base profile + strict TypeScript rules, naming conventions, and no-any enforcement. 

import strictConfig from "@ytvee-dev/eslint-config-react/configs/strict";

export default [...strictConfig];

Advanced Profile Usage

Combining Profiles

You can combine strict and React profiles for maximum control: 

import strictConfig from "@ytvee-dev/eslint-config-react/configs/strict";
import reactConfig from "@ytvee-dev/eslint-config-react/configs/react";

export default [
  ...strictConfig,  // includes base + strict rules
  ...reactConfig,   // adds React-specific rules
];

Custom Overrides

Override specific rules for your project needs: 

import reactConfig from "@ytvee-dev/eslint-config-react/configs/react";

export default [
  ...reactConfig,
  {
    rules: {
      'no-console': 'warn',
      '@typescript-eslint/explicit-function-return-type': 'off',
    },
  },
];

File-Specific Configuration

Apply different rules to test files or scripts: 

import reactConfig from "@ytvee-dev/eslint-config-react/configs/react";

export default [
  ...reactConfig,
  {
    files: ['**/*.test.{ts,tsx}'],
    rules: {
      '@typescript-eslint/no-explicit-any': 'off',
    },
  },
];

Style Guide

The configuration follows best practices from Airbnb JavaScript Style Guide with TypeScript enhancements.

Code Formatting

  • Line width: 120 characters
  • Quotes: single (')
  • Semicolons: required
  • Indentation: 2 spaces
  • Trailing comma: always

Import Organization

Imports are automatically sorted into groups:

  1. External dependencies (react, express)
  2. Namespace packages (@internal/logger)
  3. Absolute aliases (@/components/Button)
  4. Side-effect imports (./styles.css)
  5. Parent directory imports (../parent/module)
  6. Current directory imports (./helper)

Code Examples

Variables

// Good - use const for non-reassigned values
const maxItems = 100;
const userName = 'John';

// Good - use let for mutable values
let counter = 0;
counter++;

// Bad - avoid var
var oldStyle = 'deprecated';

Functions

// Good - arrow functions with implicit return
const double = (x: number): number => x * 2;

// Good - explicit return type
function calculateSum(a: number, b: number): number {
  return a + b;
}

// Good - prefer template strings
const greeting = `Hello, ${userName}!`;

Objects and Arrays

// Good - use object shorthand
const name = 'Product';
const price = 99;
const product = { name, price };

// Good - use destructuring
const { id, title } = product;
const [first, ...rest] = array;

// Good - use spread operator
const newProduct = { ...product, discount: 10 };

Comparisons

// Good - strict equality
if (value === 'test') { }
if (count !== 0) { }

// Bad - avoid == and !=
if (value == 'test') { }  // eslint error

Base Rules

The base profile includes:

ESLint Recommended

  • Code correctness checks
  • Security rules
  • Best practices enforcement

Airbnb Best Practices

  • no-var - Use const/let instead of var
  • prefer-const - Prefer const for non-reassigned variables
  • eqeqeq - Strict comparison === instead of ==
  • prefer-template - Use template strings
  • arrow-body-style - Implicit return in arrow functions

Key Rules Explained

no-var

Disallows var keyword, enforces const/let usage 

// Bad
var count = 0;

// Good
const MAX_COUNT = 100;
let count = 0;

prefer-const

Requires const for variables that are never reassigned 

// Bad
let config = { api: 'https://api.example.com' };
// config is never reassigned

// Good
const config = { api: 'https://api.example.com' };

eqeqeq

Requires strict equality operators === and !== 

// Bad
if (value == 'test') { }
if (count != 0) { }

// Good
if (value === 'test') { }
if (count !== 0) { }

prefer-template

Requires template literals instead of string concatenation 

// Bad
const message = 'Hello, ' + name + '!';

// Good
const message = `Hello, ${name}!`;

arrow-body-style

Requires concise arrow function bodies when possible 

// Bad
const double = (x) => { return x * 2; };

// Good
const double = (x) => x * 2;

object-shorthand

Requires object literal shorthand syntax 

// Bad
const obj = {
  name: name,
  getValue: function() { return 42; }
};

// Good
const obj = {
  name,
  getValue() { return 42; }
};

TypeScript Rules

TypeScript rules include type-checked mode for maximum safety:

Type Safety

  • @typescript-eslint/explicit-function-return-type - Explicit return types
  • @typescript-eslint/no-floating-promises - Handle all promises
  • @typescript-eslint/no-unused-vars - No unused variables
  • @typescript-eslint/consistent-type-definitions - Prefer interface

Class Members

  • @typescript-eslint/explicit-member-accessibility - Explicit access modifiers
  • @typescript-eslint/member-ordering - Consistent class member order

TypeScript Examples

Explicit Return Types

// Bad - no return type
function calculate(a: number, b: number) {
  return a + b;
}

// Good - explicit return type
function calculate(a: number, b: number): number {
  return a + b;
}

Promise Handling

// Bad - floating promise
fetchData();

// Good - properly handled
await fetchData();
// or
fetchData().catch(handleError);

Type Definitions

// Good - prefer interface
interface User {
  id: number;
  name: string;
}

// Use type for unions
type Status = 'active' | 'inactive' | 'pending';

Strict Rules

The strict profile adds additional constraints for large teams:

Strict TypeScript

  • @typescript-eslint/no-explicit-any - Disallow any type
  • @typescript-eslint/naming-convention - camelCase/PascalCase enforcement
  • max-classes-per-file - One public type/class per file

Strict Mode Examples

No Any Type

// Bad - any type
function process(data: any): any {
  return data.value;
}

// Good - explicit types
interface Data {
  value: string;
}

function process(data: Data): string {
  return data.value;
}

Naming Conventions

// Bad - incorrect naming
const UserName = 'John';  // variables should be camelCase
interface user_data {}    // interfaces should be PascalCase
class my_class {}         // classes should be PascalCase

// Good - proper naming
const userName = 'John';
interface UserData {}
class MyClass {}
type UserId = string;

One Export Per File

// Bad - multiple public classes
export class UserService {}
export class ProductService {}

// Good - one public export per file
// UserService.ts
export class UserService {}

// ProductService.ts
export class ProductService {}

The strict profile is recommended for large teams and enterprise projects where code consistency and type safety are critical.
